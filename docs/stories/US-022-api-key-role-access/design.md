# Design

## Domain Model

- API principal: API key id and role.
- Roles: `viewer`, `researcher`, `operator`.
- Role hierarchy: `viewer < researcher < operator`.

## Application Flow

Protected FastAPI routes use a dependency generated by `require_role`. The
dependency reads `X-API-Key`, resolves it against `AIQRA_API_KEYS`, and checks
the role hierarchy. Missing keys, invalid keys, insufficient roles, and missing
configuration fail closed.

## Interface Contract

- Public: `GET /health`.
- Requires `viewer`: `GET /metrics`, `GET /experiments`,
  `GET /experiments/{run_id}`, `GET /reports/{run_id}`,
  `GET /signals/latest`, `GET /signals/as-of`.
- Requires `researcher`: `POST /experiments/run`.

Environment:

```text
AIQRA_API_KEYS=viewer-key:viewer,researcher-key:researcher
```

## Data Model

No database migration. Keys are environment configuration only.

## UI / Platform Impact

CLI workflows are unchanged. Internal API callers must send `X-API-Key` for
non-health routes.

## Observability

Existing request logs remain. Per-request authenticated actor logging is a
follow-up because this story does not add persistent users or audit identity.

## Alternatives Considered

1. Leave the API unauthenticated.
2. Add JWT sessions and user management.
3. Protect only mutating routes.
